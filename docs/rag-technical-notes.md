# RAG 技术选型与实施笔记

> 本文档整理了张雪峰 AI Agent 项目中关于 RAG（检索增强生成）的技术讨论和决策记录。
>
> 最后更新：2026-05-30

---

## 1. RAG 是什么？

**一句话解释：让 AI 回答问题时，先从数据库里找到相关资料，再根据资料回答，而不是凭空编。**

具体到我们的项目：

```
用户问："北大计算机专业多少分？"

没有 RAG：AI 靠自己的知识回答 → 可能过时、可能编造
有了 RAG：系统先去数据库查"北京大学"+"计算机"的录取分数 → AI 根据真实数据回答
```

### 为什么需要 RAG？

| 问题 | 原因 | RAG 怎么解决 |
|------|------|--------------|
| AI 回答的分数线不准 | LLM 知识有截止日期，且会"幻觉" | 从数据库查真实数据给 AI 参考 |
| 问学校简称查不到 | 系统只支持精确匹配 | 模糊匹配 + 语义搜索 |
| 问招生政策说"暂未开放" | search_policy 工具是 stub | 接入真实政策数据 |

---

## 2. 向量检索 vs RAG 的区别

这两个概念经常被混用，但它们不是一回事：

| 概念 | 是什么 | 类比 |
|------|--------|------|
| **向量检索** | 一种搜索技术，把文本变成数字（向量），通过计算"距离"找相似内容 | 图书馆的"按主题找书"功能 |
| **RAG** | 一套完整的流程：检索相关资料 → 把资料喂给 LLM → LLM 根据资料回答 | 图书馆管理员帮你找书 + 读给你听 + 回答你的问题 |

**关系：向量检索是 RAG 的一个组件，但 RAG 不只有向量检索。**

我们的 RAG 方案包含多种检索方式：

```
用户提问
   ↓
┌─────────────────────────────────┐
│  检索层（多种方式组合）           │
│  ├── 名称模糊匹配（别名表）      │
│  ├── 全文检索（FTS5）            │
│  └── 向量检索（ChromaDB）        │
└─────────────────────────────────┘
   ↓
检索结果 + 用户问题 → LLM → 基于真实数据的回答
```

## 3. ChromaDB 是什么？为什么选它？

### ChromaDB 简介

ChromaDB 是一个轻量级的向量数据库，专门用来存储和搜索"文本的数字表示"（向量）。

```python
# 用法极其简单
import chromadb
client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_or_create_collection("schools")

# 存入数据
collection.add(
    documents=["北京大学是中国顶尖学府..."],
    ids=["school_1"],
    metadatas=[{"province": "北京", "level": "985"}]
)

# 搜索
results = collection.query(query_texts=["计算机专业好的学校"], n_results=5)
```

### 为什么选 ChromaDB？

数据量小（~3800 条），不需要上重型方案。ChromaDB 最简单，够用。

| 方案 | 优点 | 缺点 | 适合场景 | 我们的评价 |
|------|------|------|----------|------------|
| **ChromaDB** | pip install 一行搞定，零配置，数据自动持久化 | 数据量大了性能不如 FAISS | 小中型项目，快速原型 | **最合适** |
| FAISS | 速度最快，大规模首选 | 数据只在内存，重启丢失，需自己写持久化 | 百万级数据 | 杀鸡用牛刀 |
| Milvus | 功能最全，支持分布式 | 需要 Docker 部署，配置复杂 | 企业级产品 | 太重 |
| pgvector | 和 PostgreSQL 一体 | 我们用 SQLite，迁移成本大 | 已有 PostgreSQL 的项目 | 换库代价大 |
| sqlite-vec | 和现有 SQLite 集成 | 生态太新（2024 年），文档少 | 愿意尝鲜的小项目 | 稳定性优先 |

**一句话理由：数据少 + 要求简单 + 不想折腾 = ChromaDB。**

## 4. OpenAI Embedding 是什么？为什么选它？

### Embedding 简介

Embedding（嵌入）是把文本转换成一串数字的过程。这串数字能捕捉文本的"含义"，含义相近的文本，数字也相近。

```
"北京大学" → [0.12, -0.34, 0.56, ...] (1536 个数字)
"北大"     → [0.11, -0.33, 0.57, ...] (非常接近！)
"清华大学" → [0.45, 0.67, -0.12, ...] (距离较远)
```

这样就能实现语义搜索——即使用户说"北大"，也能匹配到"北京大学"的内容。

### 为什么选 OpenAI Embedding？

我们已经在用 OpenAI API，加一个嵌入模型零成本，效果还最好。

| 模型 | 维度 | 中文效果 | 调用方式 | 优缺点 |
|------|------|----------|----------|--------|
| **OpenAI text-embedding-3-small** | 1536 | 优秀 | API 调用 | 已在用 OpenAI，零配置，效果最好 |
| BAAI/bge-small-zh-v1.5 | 512 | 优秀 | 本地推理 | 免费但需下载模型（~100MB），部署多一个依赖 |
| shibing624/text2vec-base-chinese | 768 | 良好 | 本地推理 | 免费但需本地部署，模型文件大 |

**一句话理由：已经在用 OpenAI + 效果最好 + 最简单 = OpenAI Embedding。**

### 费用估算

```
数据量：~3800 条（3000 学校 + 800 专业）
OpenAI text-embedding-3-small 价格：$0.02 / 1M tokens
每条约 200 tokens → 3800 × 200 = 760K tokens
费用：760K × $0.02 / 1M = $0.015（约 1 毛钱人民币）

结论：全量嵌入一次不到 1 块钱，可以忽略不计。
```

## 5. 嵌入模型选型对比表

| 维度 | OpenAI text-embedding-3-small | BAAI/bge-small-zh-v1.5 | text2vec-base-chinese |
|------|-------------------------------|------------------------|----------------------|
| **维度** | 1536 | 512 | 768 |
| **中文效果** | 优秀 | 优秀 | 良好 |
| **英文效果** | 优秀 | 一般 | 一般 |
| **调用方式** | API（远程） | 本地推理 | 本地推理 |
| **部署复杂度** | 零（已有 OpenAI API） | 需下载模型 + PyTorch | 需下载模型 + PyTorch |
| **运行成本** | ~0.01 元/次调用 | 免费（占本地 CPU/GPU） | 免费（占本地 CPU/GPU） |
| **首次加载** | 无（API 调用） | ~5 秒（加载模型） | ~5 秒（加载模型） |
| **适用场景** | 已有 API 集成，追求简单 | 不想花钱，隐私要求高 | 不想花钱，中文为主 |

**我们的选择：OpenAI text-embedding-3-small**

理由：
1. 已经在用 OpenAI 的聊天 API，加一个嵌入模型零配置
2. 中英文效果都最好
3. 不需要下载模型文件，不占本地空间
4. 费用可以忽略不计（3800 条全量嵌入不到 1 块钱）

## 6. RAG 实施方案（5 步）

### 第 1 步：学校/专业名称模糊匹配（3 天）

**做什么：** 建一张"别名表"，存学校的简称和英文名。

```
北大 → 北京大学
清华 → 清华大学
PKU  → 北京大学
复旦 → 复旦大学
```

**做完后有什么变化：** 用户说"北大多少分"，系统能自动识别是"北京大学"。

**技术方案：**
- 新增 `school_aliases` 表
- 修改 `get_school_by_name` 查询逻辑：先查别名表 → 未命中则回退到模糊匹配
- 辅助方案：pypinyin 拼音匹配作为兜底

---

### 第 2 步：给搜索结果加缓存（2 天）

**做什么：** 把查过的数据暂时记在 Redis 里，下次问同样的问题直接从缓存取。

**做完后有什么变化：** 热门问题回答速度从 200ms 降到 20ms，体感"秒回"。

**技术方案：**
- 复用现有 Redis 连接（`backend/user_profile.py` 已有 `_get_redis()`）
- 在 CRUD 层添加缓存装饰器
- 缓存策略：学校/专业信息 TTL 24h，分数线查询 TTL 1h

---

### 第 3 步：全文搜索（3 天）

**做什么：** 用 SQLite 自带的 FTS5 功能，给学校描述、专业介绍建索引。

**做完后有什么变化：** 用户搜"计算机相关专业"、"北京 985"能搜到相关内容。

**技术方案：**
```sql
CREATE VIRTUAL TABLE schools_fts USING fts5(
    name, province, city, description,
    content='schools', content_rowid='id'
);
```
- 数据同步触发器（INSERT/UPDATE/DELETE 时自动同步）
- 新增 FTS 查询 CRUD 函数

---

### 第 4 步：补全招生政策搜索（3 天）

**做什么：** 建政策表，存 50 条核心招生政策，把 `search_policy` 从 stub 改为真实实现。

**做完后有什么变化：** 用户问"强基计划怎么报名"，AI 能基于真实政策数据回答。

**技术方案：**
- 新增 `Policy` 模型（title, content, source, province, year, category）
- 数据来源：教育部阳光高考平台、各省教育考试院
- 查询方式：短期 SQL LIKE → 中期接入 FTS5 → 长期接入向量检索

---

### 第 5 步：语义搜索（4 天）

**做什么：** 用 OpenAI 嵌入模型把描述转成向量，存在 ChromaDB 里，支持语义搜索。

**做完后有什么变化：** 用户问"适合女生的理工科专业"、"分数不高但就业好的专业"，AI 能找到相关内容。

**技术方案：**
```
数据入库 → OpenAI Embedding → 存入 ChromaDB Collection
用户查询 → OpenAI Embedding → ChromaDB 相似度搜索 → 返回结果
```

Collection 设计：
- `schools_collection`：name + description + province + level 作为嵌入文本
- `majors_collection`：name + description + job_directions + top_industries 作为嵌入文本

---

### 时间线总览

```
第1周: 第1步(名称匹配) + 第2步(缓存)        → 5天，可并行
第2周: 第3步(全文搜索) + 第4步(政策搜索)      → 6天，可并行
第3周: 第5步(语义搜索)                        → 4天
```

**总计 15 个工作日，约 3 周。**

### 依赖关系

```
名称模糊匹配 ──┐
                ├──→ 全文检索(FTS5) ──→ 补全 search_policy
查询缓存 ──────┘                         │
                                         ↓
                                    向量检索(ChromaDB)
```

## 7. 当前遇到的问题

### 嵌入 API 端点 503 错误

在测试 OpenAI Embedding API 时遇到了 503 错误：

```
openai.APIStatusError: 503 server unavailable
```

**可能原因：**
1. OpenAI API 临时过载（高峰期常见）
2. 网络代理/VPN 问题
3. API Key 权限问题

**解决方案：**
1. 添加重试逻辑（指数退避）
2. 设置合理的超时时间
3. 考虑批量嵌入时分批请求，避免并发过高
4. 长期可考虑本地嵌入模型作为降级方案

## 9. 设计原则

**核心原则：能简单就简单，够用就行，不过度设计。**

- ChromaDB > Milvus/FAISS：数据量小，不需要重型方案
- OpenAI Embedding > 本地模型：已在用 OpenAI，零额外配置
- SQLite FTS5 > Elasticsearch：数据量小，不需要独立搜索引擎
- 分阶段实施：先解决最痛的问题（名称匹配 + 缓存），再逐步增强
